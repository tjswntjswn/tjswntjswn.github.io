GitHub에서의 협업은 단순히 코드를 합치는 것이 아닙니다.
GitHub는 **'GitHub Flow'**라는 간단하고 강력한 워크플로를 제안합니다. 이는 개발자뿐만 아니라 로드맵, 문서 작업 등 다양한 분야에서 활용될 수 있습니다.

오늘은 프로젝트를 망치지 않고 안전하게 협업하는 **GitHub Flow의 6단계**를 정리해 봅니다.

> **참고 문서**: [GitHub 공식 가이드 - GitHub 흐름](https://docs.github.com/ko/get-started/using-github/github-flow)
<br>

## 1단계: 분기(Branch) 만들기
모든 작업은 **'브랜치(Branch)'**를 만드는 것에서 시작합니다.
기본 브랜치(main)에서 바로 작업하는 것은 위험합니다. 새로운 브랜치를 만들면 원본 코드에 영향을 주지 않고 안전하게 실험하고 개발할 수 있습니다.

* **Tip:** 브랜치 이름은 `increase-test-timeout`이나 `add-code-of-conduct`처럼 작업 내용을 명확히 알 수 있게 짓는 것이 좋습니다.

<br>

## 2단계: 변경하고 커밋(Commit)하기
생성된 브랜치에서 자유롭게 파일을 수정(Edit), 추가, 삭제합니다. 작업한 내용은 `Commit`을 통해 기록하고, 원격 저장소로 `Push`합니다.

### 💡 커밋 잘하는 법 (Best Practice)
1.  **설명적인 메시지:** `fix typo`(오타 수정) 처럼 변경 내용을 명확히 적으세요.
2.  **원자적 커밋(Atomic Commit):** 하나의 커밋에는 **하나의 논리적 변경**만 담는 것이 이상적입니다.
    * *나쁜 예:* 변수 이름 바꾸기 + 테스트 코드 추가를 한 번에 커밋
    * *좋은 예:* 변수 이름 변경 커밋 따로, 테스트 코드 추가 커밋 따로
    * *이유:* 나중에 문제가 생겼을 때, 테스트는 유지하고 변수 이름만 되돌리고 싶다면 커밋이 분리되어 있어야 쉽습니다.
3.  **수시로 Push:** 작업물을 원격 저장소에 백업하는 효과가 있으며, 동료들이 내 작업 진행 상황을 볼 수 있습니다.

<br>

## 3단계: 끌어오기 요청(Pull Request, PR) 만들기
작업이 어느 정도 진행되었거나, 동료의 피드백이 필요할 때 **PR(Pull Request)**을 생성합니다. 이것은 "내 코드를 기본 브랜치에 합쳐도 될까요?"라고 제안하는 과정입니다.

* **요약 작성:** 변경된 내용과 해결한 문제를 요약해서 적습니다. 이미지나 표를 활용하면 더 좋습니다.
* **이슈 연결:** `#12`와 같이 관련 이슈 번호를 언급하면 PR이 병합될 때 해당 이슈가 자동으로 닫히게 할 수 있습니다.
* **검사(Check) 실행:** 자동화된 테스트가 있다면 이 단계에서 실행되어 오류를 미리 잡아줍니다.

<br>

## 4단계: 리뷰 및 피드백 (Review)
PR을 올리면 팀원들이 코드를 검토합니다. GitHub Flow의 핵심은 **'대화'**입니다.

* **코드 리뷰:** 특정 줄에 코멘트를 달아 질문하거나 더 좋은 코드를 제안할 수 있습니다.
* **수정 반영:** 리뷰어의 피드백을 반영해 코드를 수정하고 다시 Push 하면 PR이 자동으로 업데이트됩니다.

<br>

## 5단계: 병합하기 (Merge)
리뷰가 끝나고 승인을 받았다면, 이제 변경 내용을 기본 브랜치(Main)에 **병합(Merge)**합니다.
이 과정에서 GitHub는 충돌(Conflict)이 있는지 확인해주며, 충돌이 없다면 버튼 클릭 한 번으로 코드가 합쳐집니다.

<br>

## 6단계: 브랜치 삭제하기
병합이 완료된 브랜치는 더 이상 필요하지 않습니다. 깔끔한 프로젝트 관리를 위해 브랜치를 **삭제**합니다.

* **걱정 마세요:** 브랜치를 삭제해도 커밋 기록과 PR 기록은 그대로 남습니다. 필요하다면 언제든 다시 복구하거나 되돌릴 수 있습니다.

<br>

---
### 📝 마치며
GitHub Flow는 **[ 브랜치 생성 → 커밋 → PR → 리뷰 → 병합 → 삭제 ]**로 이어지는 순환 구조입니다. 이 흐름만 익혀도 오픈소스 기여나 팀 프로젝트 협업이 훨씬 두렵지 않게 될 것입니다.
