깃허브 블로그를 만들다 보니 매번 마주치는 이름이 있다. 바로 **지킬(Jekyll)**이다.
GitHub Pages를 쓰면 자연스럽게 따라오기도 하고, 테마를 적용하거나 로컬에서 실행하려고 하면 더 자주 보게 된다.
하지만 막상 “지킬이 정확히 뭘 하는지”는 흐릿하게 알고 넘어가기 쉬운 편이다.

왜 네이버 블로그처럼 관리 화면에서 글을 쓰는 방식이 아니고,
왜 파일을 만들고 터미널에서 명령어를 실행해야 화면이 바뀌는지,
그리고 이 과정이 어떤 원리로 동작하는지 정리해 보려고 한다.

오늘 글에서는 **정적 사이트(Static Site)**와 **정적 사이트 생성기(SSG)**의 개념을 먼저 잡고,
그다음 **Jekyll이 실제로 어떤 역할을 하는지**를 구조 관점에서 설명한다.

## 1. 정적 사이트 생성기(Static Site Generator)란?

지킬을 이해하려면 먼저 **정적(Static) 웹사이트**가 무엇인지부터 정리하는 것이 좋다.
정적 사이트는 말 그대로, **서버가 사용자의 요청에 따라 “즉석에서 페이지를 만들어내지 않는” 사이트**를 말한다.

정적 사이트의 결과물은 보통 다음과 같은 파일들이다.

- index.html 같은 **HTML 파일**
- style.css 같은 **CSS 파일**
- app.js 같은 **JavaScript 파일**
- 이미지/폰트 같은 정적 리소스

즉, 사용자가 사이트에 접속하면 서버는 계산을 하기보다 **이미 만들어져 있는 파일을 그대로 전달**한다.

### 동적 사이트 vs 정적 사이트

둘의 차이는 보통 “페이지를 만드는 시점”에서 갈린다.

#### 동적 사이트 (Dynamic Site)
- 사용자가 접속할 때마다 서버가 **데이터를 조회(DB 등)**하고
- 로그인 상태, 권한, 최신 댓글, 좋아요 수 같은 정보를 반영해
- **그 시점에 맞는 HTML을 만들어서** 내려준다

예를 들어 티스토리/네이버 블로그에서는
- 글을 쓰면 DB에 저장되고
- 방문자가 들어올 때 서버가 DB에서 글/댓글/좋아요 등을 불러와
- 매번 화면을 구성한다

장점은 “실시간 기능(로그인/댓글/개인화)”이 자연스럽다는 점이고,
단점은 “서버가 해야 할 일이 많아질수록 관리와 비용이 커진다”는 점이다.

#### 정적 사이트 (Static Site)
- 글을 작성하면(또는 수정하면) 그때 **HTML을 미리 생성(build)**
- 사용자가 접속할 때는 서버가 별도 계산 없이
- 만들어진 HTML/CSS/JS 파일을 그대로 전달한다

즉, **변경이 생겼을 때만 다시 생성**하고, 평소에는 결과물을 그대로 제공하는 방식이다.

---

### 그럼 SSG는 뭘 하는가?

**Static Site Generator(SSG)**는 정적 사이트를 만들기 위해
**원본 글/데이터를 받아서 정적 파일(HTML 등)로 “빌드”해주는 도구**다.

보통 입력은 이런 형태다.

- 글: Markdown(.md)
- 설정: _config.yml
- 템플릿/레이아웃: _layouts, _includes
- 데이터: _data/*.yml (있는 경우)

그리고 출력은 이런 형태다.

- 완성된 HTML/CSS/JS가 들어 있는 **정적 결과물 폴더**
- (Jekyll의 경우 보통 _site/)

정리하면, SSG는 “블로그 엔진”이라기보다는
**문서(텍스트) + 템플릿을 조합해 웹사이트 파일을 만들어내는 빌드 도구**에 가깝다.

## 2. Jekyll은 무엇인가?

**Jekyll**은 Ruby로 만들어진 대표적인 정적 사이트 생성기(SSG)다.
GitHub Pages에서 오랫동안 기본적으로 지원해 왔기 때문에 “깃허브 블로그=지킬”로 묶여 인식되는 경우가 많다.

Jekyll이 하는 핵심 작업은 다음 한 줄로 요약할 수 있다.

> **Markdown으로 작성한 글과 HTML 템플릿을 합쳐서 정적 HTML을 만들어낸다.**

### Jekyll의 빌드 흐름(대략)

Jekyll은 보통 아래 순서로 사이트를 생성한다.

1. 프로젝트 폴더에서 _config.yml을 읽어 **기본 설정**을 파악한다
2. _posts/(또는 페이지 파일들)의 Markdown을 읽는다
3. 각 글의 Front Matter(--- 사이 메타데이터)를 읽어 **제목/날짜/카테고리 같은 변수**를 만든다
4. 해당 글에 적용할 레이아웃을 찾아 _layouts에서 불러온다
5. 레이아웃과 글 내용을 합쳐 최종 HTML을 만든다
   (이때 Liquid 템플릿 문법이 같이 처리됨)
6. 완성된 HTML을 _site/에 저장한다

GitHub Pages를 쓰면, 로컬에서 jekyll build를 실행하지 않더라도
GitHub가 푸시된 저장소를 기반으로 위 빌드 과정을 수행해 사이트를 배포한다(설정에 따라 다름).

## 3. 파일 구조로 보는 “지킬을 쓴다”는 것

지킬을 사용한다는 건 결국 **“입력 파일(원본)을 정해진 규칙대로 배치하고, 빌드해서 결과물을 만든다”**는 의미다.
자주 보는 폴더들을 역할 기준으로 정리하면 아래처럼 생각할 수 있다.

### 글(콘텐츠)
- _posts/ : 날짜+제목 규칙으로 관리되는 블로그 글
  - 예: 2025-12-29-jekyll-intro.md

### 레이아웃(틀)
- _layouts/ : 페이지 전체 구조(헤더/푸터 포함)를 정의하는 HTML 템플릿
  - 글에서 layout: post면 _layouts/post.html을 찾는다

### 공통 컴포넌트
- _includes/ : 레이아웃에서 반복해서 쓰는 조각들(헤더, 네비게이션, 푸터 등)

### 설정
- _config.yml : 사이트 전역 설정(사이트 제목, baseurl, 플러그인, permalink 규칙 등)

### 결과물
- _site/ : 빌드 결과물이 모이는 폴더
  보통 이 폴더는 **직접 수정하지 않는다** (다시 빌드하면 덮어씌워짐)

## 4. 왜 터미널 명령어가 필요한가?

네이버 블로그처럼 “글쓰기 버튼 → 게시” 방식은 보통 서버에서 즉시 저장/반영되는 구조다.
반면 지킬은 “파일 기반 + 빌드 기반”이라서 로컬에서 다음 작업이 필요하다.

- Markdown 파일을 새로 만들거나 수정한다
- Jekyll이 이를 읽어서 HTML을 다시 생성해야 한다
- 로컬에서 확인하려면 개발 서버를 띄워야 한다

대표 명령어는 아래 두 가지다.

- jekyll build
  : 정적 결과물을 _site/에 생성
- jekyll serve
  : build + 로컬 서버 실행(파일 변경 감지 시 재생성)

즉, 터미널 명령어는 “사이트를 실행한다”기보다는
**“원본을 결과물로 다시 만들어서 확인한다”**에 가깝다.

## 5. 왜 Jekyll을 쓸까? (장단점)

### 장점
1. **속도가 빠르다**
   요청 시점에 서버가 HTML을 만들지 않으므로 응답이 단순하다.
2. **운영 부담이 적다**
   DB나 백엔드 서버가 필수가 아니어서 호스팅이 간단하다.
3. **보안 리스크가 상대적으로 낮다**
   서버 로직/DB가 없으면 공격 표면이 줄어든다.
4. **버전 관리가 쉽다**
   글도 코드처럼 Git으로 관리되므로 변경 이력이 명확하다.
5. **GitHub Pages와 결합이 좋다**
   저장소에 올리면 배포까지 자연스럽게 이어진다(구성에 따라).

### 단점
1. **동적 기능이 기본으로 없다**
   댓글/로그인/검색 같은 기능은 외부 서비스를 붙이거나 별도 구현이 필요하다.
2. **빌드 개념을 이해해야 한다**
   “저장하면 바로 반영”이 아니라 “빌드해야 반영”이라는 흐름이 있다.
3. **환경 이슈가 종종 있다**
   Ruby 버전, gem, 플러그인 호환 같은 문제가 생길 수 있다(특히 로컬).
4. **규모가 커지면 빌드 시간이 늘 수 있다**
   글이 많아질수록 생성해야 할 페이지가 많아진다.

## 6. 결론

Jekyll은 “블로그 서비스”라기보다는
**글(원본)과 템플릿(레이아웃)을 조합해 정적 결과물을 만드는 빌드 도구(SSG)**다.

네이버 블로그처럼 관리 화면에서 글을 쓰는 방식이 아니라
파일을 작성하고 빌드하는 흐름을 가진 이유도,
결국 “서버에서 매번 만들지 않고, 미리 만들어서 제공하기 위한 구조”이기 때문이다.

정적 사이트 생성기의 관점에서 보면,
- Markdown은 콘텐츠(재료)
- 레이아웃은 틀(설계도)
- Jekyll은 빌드 도구(조립 과정)
- _site/는 결과물(완성품)

정도로 정리할 수 있다.

이 구조를 이해하고 나면,
터미널에서 보던 Generating... 메시지도
“원본을 결과물로 다시 만드는 과정”으로 자연스럽게 연결된다.
